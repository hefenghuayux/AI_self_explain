# 阶段 06：语音录制、ASR 与确认

## 目标

加入语音输入，确保 AI 仍只评价学生最终确认的文本。

## 范围

- 浏览器 PCM 音频采集与本地保存。
- 浏览器到后端、后端到百炼的双向 WebSocket 实时 ASR。
- 原始转写与中间转写展示。
- 学生在录音期间继续使用键盘编辑。
- 学生修改与确认。
- ASR 错误记录和恢复。

## 修改内容

- 增加 `AudioFile` 数据模型。
- 使用浏览器 Web Audio API 采集 16 kHz、单声道 PCM，并按配置校验格式、大小和时长。
- 保存音频相对路径、媒体类型、大小和 SHA-256。
- 使用 DashScope SDK 连接百炼 `fun-asr-realtime`，增加双向流式调用记录和转写确认页面。
- 分别保存 `asrTranscript` 和 `confirmedText`。

实时连接建立失败按配置重试；流中连接中断时不重放已发送 PCM，避免重复转写。失败后退回 `CAPTURING_INPUT`，保留错误记录，允许学生重新录音；不增加支持次数，也不进入 `NEED_HUMAN`。

## 测试要求

- 支持的 PCM 音频可以流式保存。
- 音频格式、大小和时长限制从配置读取。
- 文件写入失败时不创建残缺数据库记录。
- ASR 原始转写和学生确认文本分别保存。
- 学生修改后的确认文本是 AI 唯一评价输入。
- ASR 连接建立超时和服务异常按配置重试。
- 连接重试耗尽或流中断退回 `CAPTURING_INPUT`。
- ASR 失败不创建评价或支持事件。
- 中间转写不覆盖主输入框，句末转写追加到主输入框。

## 完成标准

- 学生可以实时录音、边编辑边查看转写、修改确认并进入现有教学闭环。
- 原始音频、原始转写和确认文本可以关联追溯。
- 本阶段自动测试全部通过。

## 合理性批判与不足

1. 学生确认转写能隔离 ASR 错误，但不能证明端到端语音理解准确。
2. 本地音频存储适合单机 Demo，不能直接支持多实例服务。
3. 浏览器实时 PCM 采集依赖 Web Audio API，当前 Demo 支持最新版 Chrome 和 Edge。
