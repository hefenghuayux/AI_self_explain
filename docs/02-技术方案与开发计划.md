# AI 自讲 Demo 技术方案与开发计划

## 1. 文档目的

本文档依据《00-需求与Demo范围.md》，定义 AI 自讲 Demo 的最小可运行技术方案与分阶段开发计划。《00-需求与Demo范围.md》是需求、流程、数据契约、状态转换和计数规则的唯一业务依据。

状态转换表、计数规则和数据契约是实现依据。AI 负责评价、判断评分点覆盖情况、生成反馈和提示；业务状态转换、计数、阈值及完成标签由确定性代码控制。

本方案只实现当前 Demo 闭环，不提前实现用户权限、人工派单、大规模并发、消息队列、分布式存储或动态学习规划。

## 2. 技术栈及选择理由

### 2.1 后端

- Python：负责模型调用、ASR 接入、数据处理和离线评测。
- FastAPI：提供会话、学生回答、AI 评价、音频上传，以及 `NEED_HUMAN` 触发与原因记录接口；不实现人工派单或处理流程。
- Pydantic：校验接口数据、状态枚举、模型配置和 AI 结构化输出。
- SQLAlchemy：管理题目、会话、消息、评价和支持事件的数据访问。
- Alembic：记录数据库结构变更，保证 Demo 演进过程可追踪。
- Uvicorn：运行 FastAPI 应用。

Demo 阶段由 FastAPI 负责完整流程编排。正式接入现有系统后，可根据服务边界将学生身份、门店权限、功能开关和学习结果同步到 `sgsf-agency`，Python 服务继续负责 ASR、模型评价和离线评测。该边界是后续演进方向，具体接口契约不在本阶段实现范围内。

### 2.2 前端

- Vue 3：管理多轮对话、录音、状态变化、申诉和 `NEED_HUMAN` 结果展示界面。
- Vite：提供轻量开发服务器和前端构建能力。
- TypeScript：约束会话状态、AI 评价结果、学生操作和接口返回结构。
- 原生 Fetch API：调用 FastAPI 接口并处理流式响应。
- 浏览器 `MediaRecorder` API：采集学生自讲音频。
- Element Plus：提供表单、按钮、对话框、提示和基础布局组件。
- 少量 CSS：完成 Demo 所需布局和局部样式调整，不再引入其他大型 UI 组件库。

### 2.3 数据与文件

- SQLite 保存题目、会话、评价、事件和错误记录。
- 本地目录保存原始音频。
- 数据库只保存音频相对路径、媒体类型、大小和摘要。

暂不引入 PostgreSQL、Redis、对象存储或消息队列。

### 2.4 测试

- pytest。
- FastAPI TestClient。
- TypeScript 类型检查和 Vite 生产构建检查。
- AI 与 ASR 测试替身。
- 参数化状态转换测试。
- 离线模型评测命令。

测试替身只用于测试，不作为运行时兜底服务。

## 3. 系统模块及职责

### 3.1 配置模块

- 读取环境变量和 `.env`。
- 校验必填配置及配置间约束。
- 提供业务阈值、模型、超时、重试和文件限制。
- 配置非法时停止启动并输出明确错误。

### 3.2 题目管理模块

- 录入和编辑题目。
- 校验评分点、常见错误、其他解法、分层提示和完整解析。
- 提供 AI 评价所需的题目上下文。

Demo 不实现录题人员权限。

### 3.3 会话模块

- 创建会话并保存学生初始选择。
- 维护业务状态和流程阶段。
- 暂停与恢复会话。
- 禁止终态会话继续自动操作。
- 使用版本号防止重复提交造成重复计数。

### 3.4 输入模块

- 接收人工文本和音频。
- 调用 ASR 并展示原始转写。
- 保存学生确认或修改后的文本。
- 在 AI 评价前拦截空文本。

只有 `confirmedText` 可以进入 AI 评价。

### 3.5 AI 评价模块

- 组织题目材料、历史上下文和确认文本。
- 调用模型并要求严格结构化输出。
- 校验 JSON Schema 及字段关系。
- 保存原始响应、解析结果和校验错误。
- 生成面向学生的反馈和具体提示。

AI 负责判断正确性、完整性及评分点覆盖情况。系统不额外判断学生答案含义。

### 3.6 确定性规则引擎

- 根据合法 AI 评价执行状态转换。
- 计算支持次数及连续无进展次数。
- 判断当前轮支持上限。
- 管理第一轮解析展示和第二轮切换。
- 计算 `completionType`。
- 执行申诉、暂停和终态保护规则。

### 3.7 审计与错误模块

- 保存状态转换事件、支持事件、模型调用和 ASR 调用。
- 保存配置版本、提示词版本、错误详情和操作耗时。
- 支持按会话追溯状态与计数变化原因。

### 3.8 离线评测模块

- 导入人工真值数据。
- 读取历史评价或重新运行指定模型。
- 计算三项业务指标。
- 输出逐样本结果和汇总结果。

## 5. 状态保存方式

### 5.1 SQLite 事务

一次业务操作中的会话状态、计数器、支持事件、状态转换事件和 AI 评价关联必须在同一事务中保存。任一步失败时事务回滚并记录明确错误。

### 5.2 音频文件

音频根目录由配置指定，内部结构为：

```text
<audio-storage-dir>/<session-id>/<file-id>.<extension>
```

文件写入成功后才提交数据库记录。文件写入失败时不创建残缺的输入记录。

### 5.3 暂停与恢复

允许暂停的稳定阶段：

- `WAIT_INITIAL_CHOICE`
- `CAPTURING_INPUT`
- `CONFIRMING_TEXT`
- `WAIT_STUDENT_ACTION`
- `SHOWING_FULL_SOLUTION`

`TRANSCRIBING` 和 `AI_EVALUATING` 期间等待本次调用结束后再处理暂停请求。

恢复时将 `status` 从 `PAUSED` 改为 `IN_PROGRESS`，并把 `flowStage` 恢复为 `pausedFromStage`；轮次、计数器和历史数据不变。

### 5.4 外部调用失败后的恢复点

- ASR 重试耗尽：退回 `CAPTURING_INPUT`，保存错误，允许重新上传或发起转写。
- AI 网络、超时或服务异常重试耗尽：退回 `CONFIRMING_TEXT`，保留确认文本，允许重新评价。
- 基础设施故障不进入 `NEED_HUMAN`，也不增加支持次数。

## 6. AI 调用和结构化输出

### 6.1 输入上下文

每次评价向模型提供：

- 题目内容和标准答案。
- 全部自然语言评分点。
- 常见错误及可接受的其他解法。
- 全部分层提示。
- 完整解析，并明确限制非终局反馈不得泄露完整答案。
- 当前轮次、当前支持次数及当前轮已覆盖评分点。
- 必要的历史评价和反馈。
- 本次 `confirmedText`。

人工录入的分层提示整体作为模型上下文。提示的具体内容、措辞和聚焦方向由模型决定，不设置“人工提示耗尽”分支。

### 6.2 结构化输出

使用严格 JSON Schema：

```text
correctness:
  CORRECT
  WRONG
  UNCERTAIN

completeness:
  COMPLETE
  INCOMPLETE

nextAction:
  COMPLETE
  ASK_FOCUSED_QUESTION
  GIVE_CORRECTION
  CORRECT_AND_ASK
  GIVE_HINT
  NEED_HUMAN
```

`coveredPoints` 和 `missingPoints` 使用来自当前题目原始评分点文本的动态枚举。

### 6.3 字段关系校验

系统只校验结构和字段一致性：

- `coveredPoints` 与 `missingPoints` 不得重叠。
- 两个集合合并后必须覆盖本题全部评分点。
- 返回的评分点文本必须来自本题原始评分点。
- `CORRECT + COMPLETE` 必须配合 `COMPLETE`。
- `UNCERTAIN` 必须配合 `NEED_HUMAN`。
- `NEED_HUMAN` 必须具有非空 `needHumanReason`。
- 其他组合必须符合已确认的状态转换表。
- 明确错误时，`errorEvidence.quote` 必须是学生确认文本的原文片段。
- `confidence` 必须为数字 `1`，但不参与业务判断。

### 6.4 校验失败

1. 保存原始响应和校验错误。
2. 按配置重新调用模型，并在修正指令中提供校验错误。
3. 达到配置上限后进入 `NEED_HUMAN`。
4. 保存系统生成的模型输出不合法原因。
5. 不增加支持次数。

### 6.5 提示词管理

提示词保存为版本化文本文件，不内嵌在业务函数中，例如：

```text
prompts/evaluate_explanation.md
```

每条评价保存 `promptVersion`、模型供应商和模型名称。

## 7. 确定性状态与计数规则

### 7.1 评价状态转换

- `CORRECT + COMPLETE + COMPLETE`：进入 `COMPLETED`。
- `CORRECT + INCOMPLETE + ASK_FOCUSED_QUESTION`：继续追问，不计支持。
- `WRONG + COMPLETE + GIVE_CORRECTION`：进入支持阈值检查。
- `WRONG + INCOMPLETE + CORRECT_AND_ASK`：进入支持阈值检查，只计一次。
- `UNCERTAIN`：进入 `NEED_HUMAN`。
- `nextAction = NEED_HUMAN`：进入 `NEED_HUMAN`。

### 7.2 新增评分点

系统不判断评分点是否真的被学生讲对，只使用已通过数据契约校验的 AI 输出：

```text
newPoints =
    currentEvaluation.coveredPoints
    - session.coveredPointsCurrentRound
```

- `newPoints` 非空时，`noProgressCount` 清零。
- `newPoints` 为空时，`noProgressCount` 加一。
- 每次评价后更新当前轮集合和全会话集合。
- 达到配置中的连续无进展阈值后，下一次 `ASK_FOCUSED_QUESTION` 升级为 `GIVE_HINT`。

### 7.3 第二轮初始化

第一轮完整解析展示后，学生确认看懂：

```text
round = 2
supportCountRound = 0
noProgressCount = 0
coveredPointsCurrentRound = []
solutionExposed = true
```

`supportCountTotal`、`coveredPointsAll` 以及全部自讲、评价、支持和审计记录继续保留。

### 7.4 支持阈值

所有阈值从配置读取。达到当前轮阈值时：

- `supportCountRound` 记为阈值值。
- 不发送该次局部支持。
- 不创建 `VALID` 支持事件。
- 不增加 `supportCountTotal`。
- 第一轮展示完整解析。
- 第二轮展示完整解析并进入 `STOPPED_LIMIT`。

### 7.5 提示生成

模型结合分层提示、历史反馈、当前缺失评分点、已提供支持、当前轮次和支持次数生成下一次提示。系统只控制是否允许发送以及是否触发阈值，不判断提示的语义内容。

## 8. 配置管理

### 8.1 配置来源

- `.env.example`：非敏感配置示例。
- `.env`：本地配置，不提交密钥。
- 环境变量：覆盖 `.env`。
- 提示词文件：单独版本控制。
- 题目评价材料：保存为数据库业务数据。

### 8.2 配置约束

所有业务次数、阈值、超时、重试、文件大小限制和日志轮转数量都从配置对象读取。配置缺失或非法时停止启动并打印明确错误，不提供隐式兜底值。

### 8.3 初始配置值

```dotenv
FIRST_ROUND_SUPPORT_LIMIT=6
SECOND_ROUND_SUPPORT_LIMIT=3
NO_PROGRESS_LIMIT=2

AI_REQUEST_TIMEOUT_SECONDS=60
AI_TRANSPORT_MAX_RETRIES=2
AI_SCHEMA_MAX_RETRIES=1
AI_RETRY_BACKOFF_SECONDS=1,2

ASR_REQUEST_TIMEOUT_SECONDS=60
ASR_TRANSPORT_MAX_RETRIES=2
ASR_RETRY_BACKOFF_SECONDS=1,2

AUDIO_MAX_SIZE_MIB=20
AUDIO_MAX_DURATION_SECONDS=300

DATABASE_BUSY_TIMEOUT_SECONDS=5

LOG_MAX_SIZE_MIB=10
LOG_BACKUP_COUNT=5
```

`MAX_RETRIES` 表示首次调用之外允许的重试次数。结构校验重试与传输重试分开配置。退避数组长度必须满足对应重试次数，否则启动失败。

模型和路径配置：

```dotenv
AI_PROVIDER=
AI_BASE_URL=
AI_API_KEY=
AI_MODEL=

ASR_PROVIDER=
ASR_BASE_URL=
ASR_API_KEY=
ASR_MODEL=

PROMPT_VERSION=
DATABASE_URL=
AUDIO_STORAGE_DIR=
LOG_DIR=
```

## 9. 日志、错误记录和审计

### 9.1 应用日志

使用结构化 JSON 日志，至少包含：

```text
timestamp
level
requestId
sessionId
operation
durationMs
errorType
message
```

普通日志不直接输出 API 密钥、完整音频、完整学生文本、完整标准答案或完整模型提示词。这些业务内容按需保存在数据库审计记录中。

### 9.2 错误分类

```text
CONFIG_ERROR
VALIDATION_ERROR
DATABASE_ERROR
FILE_STORAGE_ERROR
ASR_TIMEOUT
ASR_SERVICE_ERROR
AI_TIMEOUT
AI_SERVICE_ERROR
AI_SCHEMA_ERROR
AI_RELATION_ERROR
ILLEGAL_STATE_TRANSITION
DUPLICATE_OPERATION
```

错误分类用于技术审计，不改变现有业务状态集合。

### 9.3 审计要求

系统必须能够追溯：

- 请求及其触发操作。
- 操作前后状态和流程阶段。
- 触发评价的学生自讲。
- 模型原始响应及校验结果。
- 重试原因和次数。
- 是否实际发送了支持。
- 计数器变化原因。
- 展示解析或结束的原因。
- 使用的模型和提示词版本。

状态及计数变化不得只记录在普通日志中。

## 10. 测试与模型评测

### 10.1 单元测试

- 状态转换矩阵。
- 两轮支持阈值边界。
- 阈值支持不创建事件。
- `supportCountTotal` 只统计实际支持。
- 连续无进展升级。
- 第二轮重置当前轮覆盖集合。
- 完成标签计算。
- 申诉、暂停、恢复和终态保护。
- 重复请求不重复计数。

### 10.2 Schema 测试

- 合法输出。
- 非法枚举或缺少字段。
- 评分点不属于原题。
- 覆盖点与缺失点重叠或不完整。
- 动作与评价组合冲突。
- 缺少人工处理原因。
- 错误证据无法在原文找到。
- 结构重试耗尽。

### 10.3 集成测试

- 文本输入完整流程。
- 语音上传、ASR 和确认流程。
- AI 与 ASR 超时恢复。
- 数据库事务回滚。
- 音频文件写入失败。
- 暂停恢复后状态保持。
- 第一轮到第二轮完整流程。

### 10.4 端到端测试

通过浏览器验证录题、创建会话、初始选择、文本输入、录音、转写确认、AI 反馈、请求提示、完整解析、第二轮、申诉和暂停恢复。

### 10.5 离线模型评测

评测样本至少包含：

```text
question
rubric
confirmedText
truthCorrectness
truthCompleteness
```

计算：

1. 错误放行率。
2. 正确误判率。
3. 正确但不完整识别率。

报告同时输出样本数量、逐样本真值与模型结果、模型与提示词版本、结构失败次数、`NEED_HUMAN` 次数和历史评价引用。不在程序中固化指标通过阈值、最低样本量或审核规则。

## 11. 分阶段开发计划

项目分为八个阶段，每个阶段必须形成可以运行和验证的结果，经过确认后再进入下一阶段。

| 阶段 | 名称 | 可运行结果 |
|---|---|---|
| 01 | 项目骨架与配置 | 服务可启动，健康检查和配置校验通过 |
| 02 | 题目录入与查询 | 可以录入、编辑和读取完整题目材料 |
| 03 | 会话文本输入与状态骨架 | 可以创建会话、选择“会”并保存确认文本 |
| 04 | AI 结构化评价 | 可以调用真实模型并保存合法结构化评价 |
| 05 | 教学循环与确定性规则 | 纯文本模式可运行至全部业务终态 |
| 06 | 语音录制 ASR 与确认 | 语音转写经学生确认后进入现有闭环 |
| 07 | 暂停恢复日志与审计 | 会话可恢复，状态、计数和错误可追溯 |
| 08 | 系统测试与模型评测 | 可运行完整测试并生成离线评测报告 |

各阶段的目标、范围、修改内容、测试要求和完成标准见 `docs/plans/` 下对应文档。

## 12. 合理性批判与不足

1. SQLite 和本地音频存储适合单机 Demo，但不适合多实例部署，后续迁移时需要处理文件与数据库一致性。
2. 使用自然语言评分点减少录题负担，但模型必须精确返回原始文本；动态枚举可以约束格式，仍可能增加结构重试。
3. 将全部分层提示交给模型生成具体反馈实现简单，但提示层级控制弱于确定性提示索引，需要通过评测检查是否过早泄露答案。
4. 外部服务失败后保持 `IN_PROGRESS` 可以避免把技术故障归类为人工教学问题，但长时间故障可能留下较多未结束会话。
5. 八个阶段均可独立验证，但阶段 04 接入真实模型后才能检验核心价值，前三阶段主要验证工程基础和数据闭环。
