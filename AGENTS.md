# 项目协作规则
你正在协助我开发“AI 自讲 Demo”。

开始前请先读取：
1. AGENTS.md
2. docs/ 下已有设计文档
3. 当前仓库结构和 Git 状态

协作规则：
1. 不确定的问题先集中列出并询问，不要自行假设。
2. 状态转换表、计数规则和数据契约是实现依据，Mermaid 只用于展示。
3. AI 只负责评价、生成提示和复核；业务状态转换、计数和阈值必须由确定性代码控制。
4. 不写兜底函数或冗余封装，失败时保留明确错误信息。
5. 保留原有功能和注释；需要调整原设计时，先说明原因和影响，等待我确认。
6. 修改文件前，先列出准备新增或修改的准确路径。
7. 每次只完成当前指定阶段，不提前实现后续功能。
8. 方案末尾补充 2～5 点合理性批判或不足。
9. 代码完成后执行测试、检查 Git diff，只提交本次修改的代码文件，不自动推送。

## Git 提交与推送

1. 只要 Codex 修改了代码，就必须在完成本轮代码修改和必要检查后创建一次 Git 提交。
2. 仅修改文档、配置说明或其他非代码文件时，不要求自动提交，除非用户明确要求。
3. 提交前必须执行 `git status --short` 和 `git diff`，确认提交内容准确。
4. 只暂存和提交本轮由 Codex 修改的文件，不得包含用户已有的未提交改动。
5. 禁止使用 `git add .`、`git add -A` 等可能暂存无关文件的命令；应明确指定需要暂存的文件路径。
6. Codex 根据实际改动自动生成简洁、准确的中文提交信息，格式遵循 Conventional Commits，例如：
   - `feat: 增加代码解释功能`
   - `fix: 修复配置读取异常`
   - `refactor: 简化模型调用流程`
   - `test: 补充单元测试`
7. 默认不执行 `git push`。只有用户明确要求推送时，才推送当前分支。
8. 只有用户明确要求创建新分支时，才允许创建或切换分支。新分支名称默认使用 `codex/<简短任务名>`。
9. 用户明确要求“创建新分支并提交推送”时，应依次执行：创建新分支、修改和检查代码、提交、推送新分支。
10. 未经用户明确要求，不得合并、删除或强制推送分支。
11. 如果无法区分 Codex 的改动与用户已有改动，或者检查、提交、推送失败，应停止相关 Git 操作并报告具体错误。
12. 完成代码修改后，最终回复必须说明提交哈希、提交信息、当前分支以及是否推送。

## 执行进度反馈

1. 开始执行涉及多个步骤、工具调用或预计耗时较长的任务前，先简要说明当前计划。
2. 每完成一个关键阶段，主动反馈：已完成内容、正在进行内容、下一步。
3. 连续执行期间，若超过约 180 秒没有新的结果，应发送一次简短进度更新。
4. 进度反馈应使用清晰状态：`待处理`、`进行中`、`已完成`、`阻塞`。
5. 遇到阻塞时，说明已验证的事实、阻塞原因和需要我确认的事项；不要只说“还在处理”。
6. 最终回复汇总实际改动、验证结果、遗留风险和 Git 状态。
